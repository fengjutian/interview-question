## 一、认知与定义类（开场必问）

### Q1：什么是低代码？

**标准答案：**
低代码是一种通过**配置 + 少量代码**来构建应用的开发方式，本质是把**通用业务能力工程化、平台化**，减少重复编码工作。

**核心目标：**

* 提升交付效率
* 降低开发门槛
* 规范业务实现

**加分点：**

> 低代码不是替代开发，而是**让开发从写页面，升级为造系统**

---

### Q2：低代码和无代码有什么区别？

**标准答案：**

* 低代码：面向开发者，可扩展
* 无代码：面向业务人员，封闭

| 维度    | 低代码      | 无代码  |
| ----- | -------- | ---- |
| 是否写代码 | 少量       | 不写   |
| 扩展性   | 强        | 弱    |
| 适用场景  | 中后台、复杂系统 | 简单表单 |

---

## 二、核心架构类（必考）

### Q3：低代码平台的核心架构是什么？

**标准答案（一定要画结构）：**

```
Schema（协议层）
   ↓
Renderer（渲染层）
   ↓
Runtime（运行时）
```

**说明：**

* Schema：JSON 描述页面和逻辑
* Renderer：把 Schema 渲染成真实组件
* Runtime：负责状态、事件、联动

**加分点：**

> Schema 是低代码平台的“源代码”

---

### Q4：什么是 Schema？为什么不用 HTML？

**标准答案：**
Schema 是一种**结构化、可序列化、可存储的 UI 描述协议**。

不用 HTML 的原因：

* HTML 不易做 diff
* 不支持跨端
* 无法描述业务逻辑

**示例：**

```json
{
  "type": "Input",
  "props": {
    "placeholder": "请输入"
  }
}
```

## 三、编辑器实现类（高频）

### Q5：低代码平台如何实现拖拽？

**标准答案：**

1. 拖拽组件生成元数据
2. 插入 Schema Tree
3. Renderer 自动重渲染

**关键技术点：**

* 拖拽库（dnd-kit / HTML5）
* 树结构操作
* 选中态 & 辅助线

**加分点：**

> 本质是对 Schema 的 CRUD

---

### Q6：属性配置面板怎么实现？

**标准答案：**
属性面板本质是 **Schema props 的可视化编辑器**

```ts
schema.props ↔ 配置表单
```

**难点：**

* 类型系统
* 属性依赖
* 即改即生效

---

## 四、状态与联动（拉开差距）

### Q7：组件之间如何通信？

**标准答案：**

* 全局状态（store）
* 事件系统
* 表达式系统（推荐）

```js
{{ form.age > 18 }}
```

**加分点：**

> 联动应该是**声明式表达式**，而不是写 if

---

### Q8：低代码平台的状态管理怎么做？

**标准答案：**

* 页面级 state
* 组件局部 state
* 数据源 state（接口）

状态统一由 Runtime 管理，组件只消费状态。

---

## 五、扩展与插件体系（高级必问）

### Q9：低代码平台如何支持扩展？

**标准答案：**
通过**插件化架构**：

* 组件插件
* 渲染插件
* 运行时插件

```ts
registerComponent({
  type: 'Table',
  schema: {},
  render: () => {}
})
```

---

### Q10：如何保证平台不被 Schema 写死？

**标准答案：**

* 支持 JS 表达式
* 提供 escape hatch（代码逃逸）
* 插件机制兜底

---

## 六、工程与稳定性（高级）

### Q11：Schema 如何做版本兼容？

**标准答案：**

* Schema 加 version
* 提供升级函数
* 禁止破坏性升级

```json
{
  "schemaVersion": "1.0.0"
}
```

---

### Q12：低代码平台有哪些风险？

**标准答案：**

* Schema 复杂度爆炸
* 调试困难
* 不适合复杂强定制业务

**加分点：**

> 低代码适合 80% 常规业务，剩下 20% 必须写代码

---

## 七、设计题（终极）

### Q13：让你从 0 设计一个低代码平台？

**标准答案模板：**

1. 定义 Schema 协议
2. Renderer
3. 编辑器
4. Runtime
5. 插件体系
6. 权限 & 版本

你可以在任何地方收尾用这句👇：

> **低代码的本质，是用 Schema 把 UI、逻辑、状态工程化。**
