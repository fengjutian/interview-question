# 一句话总览（先把高度拉满）

> **低代码平台的扩展，本质是把平台能力拆分为稳定内核 + 可插拔插件，通过标准协议在不改内核的前提下持续演进。**

关键词：
**内核稳定 / 插件可插拔 / 协议驱动**

---

# 二、平台需要扩展哪些东西？（先分层）

| 层级  | 可扩展内容    |
| --- | -------- |
| 物料层 | 组件、区块    |
| 渲染层 | 渲染方式、布局  |
| 运行时 | 事件、联动、状态 |
| 编辑器 | 面板、工具栏   |
| 能力层 | 数据源、权限   |

> 面试官要的是：
> **“不是只扩组件，而是全链路可扩”**

---

# 三、插件化总体架构（必画脑图）

```
低代码内核（Core）
 ├─ Schema 协议
 ├─ Renderer
 ├─ Runtime
 ├─ 生命周期
 │
 └─ Plugin Host（插件宿主）
      ├─ 组件插件
      ├─ 编辑器插件
      ├─ 运行时插件
```

---

# 四、插件的核心设计（重点）

## 1️⃣ 插件统一协议（非常关键）

```ts
interface LowCodePlugin {
  name: string
  version: string
  apply(ctx: PluginContext): void
}
```

> 面试关键词：
> **统一接口 / 生命周期 / 上下文注入**

---

## 2️⃣ 插件能做什么？（通过 ctx）

```ts
ctx.registerComponent()
ctx.registerPanel()
ctx.on('schema:change', handler)
```

**核心原则：**

> 插件永远通过 **API** 操作平台，而不是直接改内核

---

## 3️⃣ 插件生命周期（一定要说）

```ts
onInit → onMount → onDispose
```

| 生命周期      | 作用   |
| --------- | ---- |
| onInit    | 注册能力 |
| onMount   | 开始生效 |
| onDispose | 清理资源 |

---

# 五、几种常见插件类型（逐个说）

## 1️⃣ 组件插件（最常见）

```ts
registerComponent({
  type: 'MyTable',
  render,
  propsSchema
})
```

用途：

* 扩展物料库
* 支持业务组件

---

## 2️⃣ 编辑器插件（高级）

> 面试官一听这个就知道你不是“只写页面的”

* 自定义属性面板
* 顶部工具栏
* 右键菜单

```ts
ctx.registerEditorPlugin()
```

---

## 3️⃣ 运行时插件（最值钱）

* 表达式引擎
* 权限系统
* 埋点 / 日志

```ts
ctx.on('component:click', ...)
```

---

# 六、插件加载与隔离（必追问）

## Q1：插件怎么加载？

**标准答案：**

* 配置化加载
* 动态 import
* 版本控制

```ts
await import('plugin-xxx')
```

---

## Q2：插件会不会把平台搞挂？

> 必问 ❗

**高分回答：**

* 沙箱执行
* try/catch 包裹
* 错误隔离 + 降级

```ts
safeApply(plugin)
```

---

## Q3：插件之间冲突怎么办？

**稳答：**

* 命名空间隔离
* 优先级机制
* 能力声明

# 七、插件与 Schema 的关系（高级）

> 面试官很爱追这一层

### 原则：

* 插件 **不直接修改 Schema**
* 插件注册 **Schema 能力**

```ts
ctx.registerSchemaEnhancer()
```

# 九、低代码扩展的最大风险（加分）

**真实回答：**

* 插件过多 → 平台失控
* 插件 API 不稳定 → 生态崩

**解决方案：**

> 内核极简 + API 稳定 + 插件强约束
